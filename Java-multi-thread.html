多线程编程

哲学家进餐问题

5个哲学家吃饭， 坐在一张圆桌旁。它们有5根筷子，每2个人中间放一根筷子。哲学家们时而思考，时而进餐。每个人都需要一双筷子才能吃到东西，并在吃完后将筷子放回原处继续思考。

由此可能产生的问题：
当所有人都已经先拿起了左手的筷子，然后尝试拿起右手的筷子，但由于此时已经没有空闲的筷子，所有人都会陷入无尽的等待中，一直吃不到东西。

 

这就是经典的"哲学家进餐"问题，由此引出多线程死锁问题。

死锁的分类

锁顺序死锁

public class LeftRightDeadLock{
  private final Object left  = new Object();
  private final Object right = new Object();

  public void leftRight() {
      synchronized(left) {
          synchronized(right) {
              doSomething();
          }
      }
  }

  public void rightLeft() {
      synchronized(right) {
          synchronized (left) {
              doSomethingElse();
          }
      }
  }
}

 

当两个线程分别呼叫leftRight()和rightLeft()方法时，就有可能产生死锁。

st=>start: A op1=>operation: 锁住left op2=>operation: 尝试锁住 right op3=>operation: 永久等待 st(right)->op1(right)-->op2(right)->op3
st=>start: B op1=>operation: 锁住right op2=>operation: 尝试锁住left op3=>operation: 永久等待 st(right)->op1(right)->op2(right)->op3

 
这类多个线程试图以不同顺序来获取相同锁造成的死锁，就是锁顺序死锁。

线程死锁的本质原因都是如此，但是由于表现形式不同或者发生竞争的资源不同又可细分为以下几类：

动态的锁顺序死锁：

没有固定的锁顺序，只有在运行期才能决定。

public void transferMoney(Account from,
                          Account to,
                          long money) {
    synchronized(from) {
        synchronized(to) {
            from.debit(money);
            to.credit(money);
        }
    }
}

一旦发生两个账户同时向对方转账，就有可能会发生死锁:

Thread A: transferMoney(alice, bob, 100);
Thread B: transferMoney(bob, alice, 300);

协作对象间的死锁：

在单独代码块中只会持有一把锁，从局部看来并不会发生死锁的风险。但是多个这样的代码块互相调用后，发生了顺序锁死锁。

  • 在持有锁的情况下调用某个外部方法，要非常警惕，仔细确认调用流程中是否也包含加锁过程，如果包含就要仔细分析避免发生死锁。

避免死锁的方法

如果一个程序每次至多获取一个锁，就不会产生锁顺序死锁。

在哲学家进餐问题中，相当于用叉子代替筷子，无论如何拿都早晚可以吃到东西。
 

  • 显而易见这样就不会出现永远吃不到东西的情况，一劳永逸的解决死锁问题。
  • 但在很多情况下这种方法并不现实，在实际工作中经常遇到必须同时持有多把锁的情况。

采用定时锁

哲学家进餐问题中，如果有人拿到一只筷子，等待一段时间发现另一只筷子始终被别人占用，那么就放下自己手中的筷子，过几分钟后再次尝试。
 

  • 这种方式可以较为简单粗暴的避免锁顺序死锁
  • 但这种方式有可能引入性能问题
    • 在放下自己手中筷子等待重试的这个过程会引入额外的等待时间。
    • 在某些更糟糕的情况下会发生活锁，即所有人同时拿起一个筷子，发现另一只已经被占用后同时放下手中的筷子，然后过几分钟后又同时拿起筷子，如此往复。
      • 这种情况下等待时间又需要引入随机性，避免活锁发生。

在必须获取多个锁的情况下，可以采用按照固定顺序加锁的方法，避免锁顺序死锁

哲学家进餐问题中，相当于给全部筷子从1到5编号， 然后规定所有科学家都必须按从小到大的顺序拿起筷子，并按相反方向放下筷子，这样也可以保证不会出现有人永远吃不到的情况。

 
  • 这是解决锁顺序死锁问题最常见的方法，无论同时申请多少锁资源都不会发生死锁问题。
  • 在某些情况下也会存在性能问题，比如将规则改为科学家可以拿筷子的数量不限，某个人已经拿了4，5后又突然想拿2。那么他就必须先放下5，再放下4，再尝试拿2，从头开始资源竞争，这样效率并不高

引入服务生

在哲学家餐桌旁增加一名服务生，这名服务生了解所有筷子的使用情况。当某个哲学家想要吃东西时，由服务生进行调配，这样可以避免有人永远吃不到食物。

  • 这种方式也可以很好的解决死锁问题，并且可以灵活的制定策略。
  • 但这种方式也需要引入额外的模块开销，增加系统复杂度。
